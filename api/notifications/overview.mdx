<Warning>
  **Cambio de contrato (breaking):** los endpoints de creación ahora reciben **`idLead`** en lugar de `idActividad`. Además, **`idTemplate` pasó de obligatorio a opcional/nullable** tanto en recordatorios como en ambos correos del seguimiento.
</Warning>

<Warning>
  **Breaking (recordatorios):** `POST /notifications/reminders` ya **no** acepta `fechaEnvio` (fecha absoluta). En su lugar recibe **`minutosAntes`** (entero `1–120`): minutos antes de que finalice la actividad en que se envía el recordatorio (máximo 2 horas antes).
</Warning>

<Warning>
  **Breaking (Calendario, CU007):** el endpoint `POST /activities` ya **no** acepta `syncWithMicrosoft` ni `createTeamsMeeting`. El evento de Outlook/Teams **ya no se crea automáticamente** al registrar la actividad; ahora se crea **bajo demanda** con [`POST /activities/{id}/calendar-event`](/api/activities/overview#post-activities-id-calendar-event).
</Warning>

El módulo de Notificaciones permite **programar, consultar y cancelar** correos asociados al seguimiento de leads y actividades. Hay dos tipos: **Recordatorio** (un correo interno al encargado de la actividad) y **Seguimiento** (correo interno al encargado \+ correo externo al contacto del lead).

**Modelo clave:** al crear una notificación se envía el **`idLead`**, no la actividad. El sistema resuelve automáticamente la **única actividad activa** del lead (regla de negocio: un lead tiene como máximo una actividad activa) y de ella obtiene la actividad, el encargado y la fecha fin. Los datos de actividad/encargado que muestre la interfaz son **solo informativos**.

**Plantilla opcional:** se puede seleccionar una plantilla activa para precargar asunto y cuerpo, pero no es obligatoria. El asunto y el cuerpo siempre se envían como **copia editable** (vengan de una plantilla o escritos a mano); la plantilla original nunca se modifica.

Las notificaciones **no se reprograman**: para cambiar fecha, hora o plantilla, cancela la existente y crea una nueva.

**Authentication:** Bearer JWT required (`Authorization: Bearer <token>`). See [Authentication](/authentication) for how to obtain and pass the token.

**OpenAPI tag:** `notifications`

## POST /notifications/reminders — Crear recordatorio

Programa un correo interno al encargado de la actividad activa del lead.

### Body parameters

<ParamField body="idLead" type="integer ≥ 1" required>
  El servidor resuelve la actividad activa del lead.
</ParamField>

<ParamField body="minutosAntes" type="integer (1–120)" required>
  Minutos antes de que finalice la actividad en que se envía el recordatorio. Rango permitido **`1–120`** (máximo 2 horas antes). El servidor calcula la fecha/hora de envío como `fechaFin de la actividad − minutosAntes`.
</ParamField>

<ParamField body="idTemplate" type="integer ≥ 1 | null">
  **Opcional.** Si se omite o es `null`, no se valida plantilla.
</ParamField>

<ParamField body="asunto" type="string (1–255)" required>
  Copia editable.
</ParamField>

<ParamField body="cuerpo" type="string" required>
  Copia editable (HTML o texto). No vacío.
</ParamField>

### Ejemplo request (con plantilla)

```json
{
  "idLead": 12,
  "minutosAntes": 30,
  "idTemplate": 5,
  "asunto": "Recordatorio: actividad pendiente",
  "cuerpo": "<p>Recuerda revisar la actividad asociada al lead.</p>"
}
```

### Ejemplo request (sin plantilla)

```json
{
  "idLead": 12,
  "minutosAntes": 60,
  "idTemplate": null,
  "asunto": "Recordatorio: actividad pendiente",
  "cuerpo": "Recuerda revisar la actividad asociada al lead."
}
```

### Respuesta — `201 Created`

Objeto `NotificationResponseDto` (ver [esquema](#esquema-notificationresponsedto)).

```json
{
  "id": 101,
  "tipo": "RECORDATORIO",
  "estado": "PROGRAMADA",
  "idActividad": 34,
  "correoCliente": null,
  "asuntoExterno": null,
  "fechaEnvioExterno": null,
  "enviadoExterno": false,
  "createdAt": "2026-06-14T20:05:00.000Z"
}
```

**Errores posibles:** `401` (sin token), `404` lead sin actividad activa, `409` notificación duplicada, `404` plantilla inexistente/inactiva, `400` `minutosAntes` fuera de `1–120`, `400` el envío calculado queda **en el pasado** porque la actividad finaliza demasiado pronto. Ver [tabla de errores](#errores).

## POST /notifications/follow-ups — Crear seguimiento

Programa un **seguimiento** con una sola instancia: primero envía un correo **interno** al encargado de la actividad y luego uno **externo** al contacto del lead.

### Body parameters

<ParamField body="idLead" type="integer ≥ 1" required>
  El servidor resuelve la **actividad activa única** del lead.
</ParamField>

<ParamField body="correoCliente" type="string (email)" required>
  Destinatario del cliente para el correo externo. Debe ser uno de los correos del contacto del lead.
</ParamField>

<ParamField body="instancias" type="array (exactamente 1 elemento)" required>
  Debe contener **exactamente 1 elemento**. Cada elemento es un objeto `{ internal, external }` con los siguientes campos:

  | Campo | Tipo | Obligatorio | Notas |
  | --- | --- | --- | --- |
  | `internal.fechaEnvio` | string (ISO 8601) | sí | Instante UTC — se respeta tal cual |
  | `internal.idTemplate` | integer ≥ 1 \| null | **no** | Default `null`. Si se omite o es `null`, asunto y cuerpo van escritos a mano |
  | `internal.asunto` | string (1–255) | sí |  |
  | `internal.cuerpo` | string (HTML) | sí |  |
  | `external.fechaEnvio` | string (ISO 8601) | sí | **Debe ser posterior a `internal.fechaEnvio`** |
  | `external.idTemplate` | integer ≥ 1 \| null | **no** | Default `null`. Si se omite o es `null`, asunto y cuerpo van escritos a mano |
  | `external.asunto` | string (1–255) | sí |  |
  | `external.cuerpo` | string (HTML) | sí |  |
</ParamField>

<Note>
  **Reglas de validación**

  - `instancias` debe tener **exactamente 1 elemento**.
  - El correo **interno** (al responsable) debe ser **anterior** al **externo** (al cliente).
  - **Todas** las fechas (interna y externa) deben ser futuras, interpretadas como instantes UTC, y caer **antes de la fecha de fin** de la actividad activa del lead.
</Note>

### Ejemplo request

```json
{
  "idLead": 12,
  "correoCliente": "maria.torres@cliente.com",
  "instancias": [
    {
      "internal": {
        "fechaEnvio": "2026-06-20T14:00:00.000Z",
        "idTemplate": 5,
        "asunto": "Revisión interna de seguimiento",
        "cuerpo": "<p>Revisa la propuesta enviada al lead.</p>"
      },
      "external": {
        "fechaEnvio": "2026-06-20T16:00:00.000Z",
        "idTemplate": null,
        "asunto": "Seguimiento a su propuesta",
        "cuerpo": "<p>Estimada María, damos seguimiento a la propuesta enviada.</p>"
      }
    }
  ]
}
```

### Respuesta — `201 Created`

`NotificationResponseDto` con `tipo: "SEGUIMIENTO"` y el array `instancias` poblado.

```json
{
  "id": 102,
  "tipo": "SEGUIMIENTO",
  "estado": "PROGRAMADA",
  "idActividad": 34,
  "idLead": 12,
  "idResponsable": 7,
  "correoCliente": "maria.torres@cliente.com",
  "instancias": [
    {
      "id": 501,
      "orden": 1,
      "asuntoInterno": "Revisión interna de seguimiento",
      "fechaEnvioInterno": "2026-06-20T14:00:00.000Z",
      "enviadoInterno": false,
      "asuntoExterno": "Seguimiento a su propuesta",
      "fechaEnvioExterno": "2026-06-20T16:00:00.000Z",
      "enviadoExterno": false
    }
  ],
  "createdAt": "2026-06-14T20:06:00.000Z"
}
```

### Errores

- **`409 Conflict`** si el lead **ya tiene un seguimiento programado**.
- **`404 NotFound`** si el lead **no tiene actividad activa**.
- **`400 Validation`** si `correoCliente` **no pertenece** a los contactos del lead.
- **`400 Validation`** si las fechas **no cumplen el rango/orden**: `instancias` con cantidad distinta de 1, `external.fechaEnvio` ≤ `internal.fechaEnvio`, o cualquier fecha en/después de la fecha de fin de la actividad.

Ver [tabla de errores](#errores) para el envelope completo.

## PATCH /notifications/follow-ups/:id — Editar seguimiento

Edita la única instancia de un seguimiento. Solo se permite cuando la notificación está **PROGRAMADA** y sus dos correos (**interno** y **externo**) todavía no han sido enviados.

Al editar, el sistema reemplaza asunto/cuerpo/plantilla/fecha de ambos correos, cancela los envíos programados anteriores y reprograma con las nuevas fechas.

<Note>
  A diferencia de `POST /notifications/follow-ups`, este endpoint **no** usa el array `instancias`. El body usa directamente los objetos planos `internal` y `external`.
</Note>

### Path parameters

<ParamField path="id" type="number" required>
  ID de la notificación de seguimiento.
</ParamField>

### Body parameters

<ParamField body="correoCliente" type="string (email)">
  Opcional. Si se omite, se conserva el correo actual. Si se envía, debe pertenecer al contacto del lead.
</ParamField>

<ParamField body="internal" type="object" required>
  Correo interno al responsable.
</ParamField>

<ParamField body="external" type="object" required>
  Correo externo al cliente. Debe programarse después de `internal`.
</ParamField>

Campos de `internal` y `external`:

| Campo | Tipo | Obligatorio | Notas |
| --- | --- | --- | --- |
| `fechaEnvio` | string (ISO 8601) | sí | Ejemplo: `2026-06-20T14:00:00.000Z` |
| `idTemplate` | integer ≥ 1 \| null | no | Default `null`. Si se omite o es `null`, se usa contenido manual |
| `asunto` | string (1–255) | sí |  |
| `cuerpo` | string (HTML) | sí |  |

<Note>
  **Reglas de validación**

  - Solo editable mientras el seguimiento esté en estado `PROGRAMADA`.
  - Ninguno de los dos correos de la instancia debe haber sido enviado.
  - `external.fechaEnvio` debe ser posterior a `internal.fechaEnvio`.
  - Ambas fechas deben ser futuras, interpretadas como instantes UTC, y anteriores a la fecha de fin de la actividad activa.
</Note>

### Ejemplo request

```json
{
  "correoCliente": "cliente@empresa.com",
  "internal": {
    "fechaEnvio": "2026-06-20T14:00:00.000Z",
    "idTemplate": 5,
    "asunto": "Recordatorio interno actualizado",
    "cuerpo": "<p>Revisar la actividad del lead</p>"
  },
  "external": {
    "fechaEnvio": "2026-06-20T15:00:00.000Z",
    "idTemplate": 6,
    "asunto": "Seguimiento de su solicitud",
    "cuerpo": "<p>Buen día, hacemos seguimiento…</p>"
  }
}
```

### Respuesta — `200 OK`

`NotificationResponseDto` con `tipo: "SEGUIMIENTO"` e `instancias` con un único elemento.

```json
{
  "id": 20,
  "tipo": "SEGUIMIENTO",
  "estado": "PROGRAMADA",
  "idActividad": 1,
  "idLead": 2,
  "idResponsable": 3,
  "asuntoInterno": null,
  "fechaEnvioInterno": null,
  "enviadoInterno": false,
  "correoCliente": "cliente@empresa.com",
  "createdAt": "2026-06-18T10:00:00.000Z",
  "instancias": [
    {
      "id": 100,
      "orden": 1,
      "asuntoInterno": "Recordatorio interno actualizado",
      "fechaEnvioInterno": "2026-06-20T14:00:00.000Z",
      "enviadoInterno": false,
      "asuntoExterno": "Seguimiento de su solicitud",
      "fechaEnvioExterno": "2026-06-20T15:00:00.000Z",
      "enviadoExterno": false
    }
  ]
}
```

### Errores

- **`404 NotFound`** si la notificación no existe.
- **`409 Conflict`** (`FollowUpNotEditable`) si no es un seguimiento, no está en `PROGRAMADA`, o ya se envió alguno de los dos correos.
- **`400/409`** si el lead no tiene actividad activa, `correoCliente` no pertenece al contacto, la plantilla no existe/está inactiva o las fechas violan las reglas de validación.

Ver [tabla de errores](#errores) para el envelope completo.

## GET /notifications — Listar notificaciones

Lista las notificaciones en **Programadas** y **Vencidas**. Las canceladas **nunca** se devuelven.

### Query parameters

<ParamField query="estado" type="string">
  `PROGRAMADA` | `VENCIDA`. Solo estos dos valores; otro → `400`.
</ParamField>

<ParamField query="idLead" type="integer ≥ 1">
  Filtra por lead.
</ParamField>

<ParamField query="idResponsable" type="integer ≥ 1">
  Filtra por encargado.
</ParamField>

<ParamField query="page" default="1" type="integer ≥ 1">
  Número de página.
</ParamField>

<ParamField query="limit" default="10" type="integer ≥ 1">
  Resultados por página.
</ParamField>

### Respuesta — `200 OK`

Objeto paginado con los resultados en `data` y metadatos en `meta`. Los items se ordenan por `fechaEnvioInterno` ascendente.

```json
{
  "data": [
    {
      "id": 101,
      "tipo": "RECORDATORIO",
      "estado": "PROGRAMADA",
      "idActividad": 34,
      "idLead": 12,
      "idResponsable": 7,
      "asuntoInterno": "Recordatorio: actividad pendiente",
      "fechaEnvioInterno": "2026-06-20T14:00:00.000Z",
      "enviadoInterno": false,
      "correoCliente": null,
      "asuntoExterno": null,
      "fechaEnvioExterno": null,
      "enviadoExterno": false,
      "createdAt": "2026-06-14T20:05:00.000Z"
    }
  ],
  "meta": {
    "page": 1,
    "limit": 10,
    "total": 1,
    "totalPages": 1
  }
}
```

## DELETE /notifications/:id — Cancelar notificación

Cancela una notificación **PROGRAMADA** (anula sus envíos). Solo aplica a notificaciones programadas.

- **Path param:** `id` (integer).
- **Respuesta `200`:** `NotificationResponseDto` con `estado: "CANCELADA"`.
- **Error `409`** (`kind: Conflict`) si la notificación no está en estado `PROGRAMADA`.

### Ejemplo request

```bash
curl -X DELETE \
  'https://api.bioactiva.com/notifications/101' \
  -H 'Authorization: Bearer <token>'
```

## Errores

Todas las respuestas de error usan el envelope:

```json
{
  "statusCode": 404,
  "error": "LeadHasNoActiveActivityException",
  "kind": "NotFound",
  "message": "El lead con id 12 no tiene una actividad activa",
  "path": "/notifications/reminders",
  "timestamp": "2026-06-14T20:05:00.000Z"
}
```

| statusCode | kind | Cuándo |
| --- | --- | --- |
| `400` | `Validation` | Body inválido, fecha fuera de rango, externo no posterior al interno, `correoCliente` no asociado al lead, `minutosAntes` fuera de `1–120`, o el envío calculado del recordatorio queda en el pasado |
| `401` | `Unauthorized` | Falta o es inválido el token Bearer |
| `404` | `NotFound` | **Lead sin actividad activa** (nuevo), o `idTemplate` enviado pero plantilla inexistente/inactiva |
| `409` | `Conflict` | Ya existe una notificación `PROGRAMADA` para esa actividad, o se cancela una no programada |

Ver [Errors](/errors) para el formato completo del envelope.
