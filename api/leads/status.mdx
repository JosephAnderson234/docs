---
title: "PATCH /leads/:id/status — Change a Lead's Pipeline Status"
sidebarTitle: "Change Status"
description: "Move a lead through the sales pipeline with enforced state transitions and conflict guards."
---

Use this endpoint to change a lead status manually, following the backend state machine rules. The backend keeps the lead and its linked quotation in sync bidirectionally — quotation outcomes move the lead (see the quotation lifecycle), and lead status changes also cascade to the linked quotation (see Lead–Quotation synchronization below).

---

## Endpoint

```text
PATCH /leads/:id/status
```

**Authentication:** JWT required.

---

## Path parameter

<ParamField path="id" type="number" required>
  Numeric ID of the lead whose status you want to change.
</ParamField>

---

## Request body

<ParamField body="estado" type="string" required>
  The new pipeline status for this lead. Must be one of the following valid enum values:

  - `EN_PROSPECTO`
  - `OFERTADO`
  - `CIERRE_CON_VENTA`
  - `CIERRE_SIN_VENTA`
</ParamField>

### Example

```json
{ "estado": "OFERTADO" }
```

---

## Response — `200 OK`

Returns the fully updated lead object, identical in structure to the response from `GET /leads/:id`. The `ultimoCambioEstado` field reflects the timestamp of the status change.

---

## Errors

| Status | `error` | When |
| --- | --- | --- |
| `400` | — | `estado` is not a valid enum value |
| `404` | — | Lead not found |
| `409` | `CONFLICT` | The status transition is invalid for the current state **or** the lead has pending activities that block the transition. |

### `409 Conflict` examples

Invalid transition:

```json
{
  "statusCode": 409,
  "error": "CONFLICT",
  "message": "Transición de estado inválida para el lead 12"
}
```

Pending-activity guard:

```json
{
  "statusCode": 409,
  "error": "CONFLICT",
  "message": "No se puede cambiar el estado del lead 12 porque tiene actividades pendientes"
}
```

---

## Transition rules

The lead can only move with these valid transitions:

| Current state | Allowed target state(s) |
| --- | --- |
| `EN_PROSPECTO` | `OFERTADO` |
| `OFERTADO` | `CIERRE_CON_VENTA`, `CIERRE_SIN_VENTA` |
| `CIERRE_CON_VENTA` | `OFERTADO`, `CIERRE_SIN_VENTA` |
| `CIERRE_SIN_VENTA` | `OFERTADO`, `CIERRE_CON_VENTA` |

Additional rules:

- Once a lead leaves `EN_PROSPECTO`, it cannot return to `EN_PROSPECTO`.
- Sending the same current status is a valid no-op and does not fail.
- Attempting to close directly from `EN_PROSPECTO` returns `409 Conflict`.

```text
EN_PROSPECTO
    │
    ▼
OFERTADO
  ├──► CIERRE_CON_VENTA
  └──► CIERRE_SIN_VENTA

CIERRE_CON_VENTA ─────────► OFERTADO
CIERRE_SIN_VENTA ─────────► OFERTADO
CIERRE_CON_VENTA ◄────────► CIERRE_SIN_VENTA
```

---

## Pending-activity guard

Any real transition is blocked while the lead still has unresolved pending activities. Resolve pending activities first (complete or cancel), then retry `PATCH /leads/:id/status`.

<Tip>
  Re-sending the same status (no-op) does not trigger this guard because there is no state transition.
</Tip>

## Lead–Quotation synchronization

The lead and its linked quotation (1:1 relationship) stay synchronized automatically. Synchronization works in both directions.

### Quotation → Lead

When the quotation lifecycle endpoints are used, the linked lead is updated atomically in the same database transaction:

| Quotation action | Effect on lead |
|---|---|
| `PATCH /quotations/:id/accept` → `ACEPTADA` | Lead moves to `CIERRE_CON_VENTA` |
| `PATCH /quotations/:id/reject` → `RECHAZADA` | Lead moves to `CIERRE_SIN_VENTA` |

### Lead → Quotation

When you change the lead status via this endpoint, the linked quotation updates accordingly:

| Lead status change | Effect on linked quotation |
|---|---|
| → `OFERTADO` | If no quotation exists, a draft `PENDIENTE` quotation is auto-generated. If a quotation already exists (e.g., reopening from a closed state), it is reset to `PENDIENTE`. |
| → `CIERRE_CON_VENTA` | Quotation moves to `ACEPTADA`. |
| → `CIERRE_SIN_VENTA` | Quotation moves to `RECHAZADA`. |

### State mapping

| Lead status | Quotation status |
|---|---|
| `OFERTADO` | `PENDIENTE` |
| `CIERRE_CON_VENTA` | `ACEPTADA` |
| `CIERRE_SIN_VENTA` | `RECHAZADA` |

Lead statuses without an entry above (e.g. `EN_PROSPECTO`) have no corresponding quotation mapping.

### Best-effort and guard bypass

The lead→quotation sync is **best-effort**:

- If the lead has no linked quotation (e.g. `AUTO_COTIZACION_ON_OFERTADO=false`), the lead status change still completes.
- If synchronization fails for any reason, the lead status change is **not** reverted.

Unlike quotation-driven transitions (accept/reject), the lead→quotation sync does **not** enforce the normal quotation state machine guards. Since the lead is the source of truth when the change originates there, a quotation in `ACEPTADA` can be moved directly to `RECHAZADA` if the lead transitions from `CIERRE_CON_VENTA` to `CIERRE_SIN_VENTA`, and vice versa.

<Note>
  **Auto-generated draft on `OFERTADO`** — The draft quotation is created in `PENDIENTE` and pre-filled from the lead and related entities:

  | Quotation field | Source |
  | --- | --- |
  | `nombreServicio` | Lead's `servicioInteres`. |
  | Cliente / dirigido | The lead's organization details (name, RUC, address, etc.). |
  | Remitente | The lead's encargado (assigned user). |
  | `monto` | Initial value `"0.00"` — editable afterwards. |
  | `tipo` (moneda) | `PEN` — editable afterwards. |

  **Behavior rules:**

  - **Real transition only.** Generation only runs when `estado` actually changes to `OFERTADO`.
  - **No duplicates.** If the lead already has a quotation, no new draft is created; instead, the existing quotation is reset to `PENDIENTE`.
  - **Best-effort.** If draft generation fails, the lead status change is not reverted.
  - **Configurable.** Controlled by `AUTO_COTIZACION_ON_OFERTADO` (enabled by default, disabled with `"false"`).
</Note>

## `ultimoCambioEstado`

Every successful status change updates `ultimoCambioEstado` to the current UTC timestamp.
