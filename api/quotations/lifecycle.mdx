Three dedicated endpoints manage the quotation lifecycle. You use them to move a quotation from its initial `PENDIENTE` state through to a final outcome. Crucially, **accepting or rejecting a quotation also closes the linked lead in the same atomic transaction** ŌĆö if the database write fails for any reason, neither the quotation nor the lead will change state.

---

## State machine

```
PENDIENTE
  ŌööŌöĆŌöĆ /send ŌåÆ ENVIADA
        Ōö£ŌöĆŌöĆ /accept ŌåÆ ACEPTADA (lead ŌåÆ CIERRE_CON_VENTA)
        ŌööŌöĆŌöĆ /reject ŌåÆ RECHAZADA (lead ŌåÆ CIERRE_SIN_VENTA)
PENDIENTE
  Ōö£ŌöĆŌöĆ /accept ŌåÆ ACEPTADA (lead ŌåÆ CIERRE_CON_VENTA)
  ŌööŌöĆŌöĆ /reject ŌåÆ RECHAZADA (lead ŌåÆ CIERRE_SIN_VENTA)
```

`ACEPTADA` and `RECHAZADA` are **terminal states** for the quotation lifecycle endpoints. Once a quotation reaches either state, no further edits or status transitions are allowed via the quotation API.

<Note>
  Synchronization is bidirectional. When the linked lead status changes via `PATCH /leads/:id/status`, the quotation status is updated automatically ŌĆö see the [lead status change](/api/leads/status#lead-quotation-synchronization) docs for the complete state mapping.
</Note>

## PATCH /quotations/:id/send ŌĆö Mark as sent

Sets the quotation status to `ENVIADA`. Only allowed when the quotation is currently in `PENDIENTE`.

**Authentication**: JWT required.

**Request**: No body required.

**Response `200 OK`** ŌĆö Updated quotation object.

**Errors**

| Status | When |
|---|---|
| 400 | Quotation is not in `PENDIENTE` state |
| 404 | Quotation not found |

---

## PATCH /quotations/:id/accept ŌĆö Accept the quotation

Sets the quotation to `ACEPTADA` and simultaneously transitions the linked lead to `CIERRE_CON_VENTA`. Allowed from either `PENDIENTE` or `ENVIADA`.

**Authentication**: JWT required.

**Request**: No body required.

**Response `200 OK`** ŌĆö Updated quotation object.

**Errors**

| Status | When |
|---|---|
| 400 | Quotation is not in `PENDIENTE` or `ENVIADA` |
| 404 | Quotation or lead not found |
| 409 | Two requests tried to finalize the same quotation at the same time ŌĆö one succeeded and this one was rejected |
| 409 | The linked lead has unresolved pending activities ŌĆö resolve them before accepting. The quotation and lead are not changed. |

**`409 Conflict`** ŌĆö lead has pending activities:

```json
{
  "statusCode": 409,
  "error": "LeadHasPendingActivitiesException",
  "kind": "Conflict",
  "message": "No se puede aceptar la cotizaci├│n porque el lead 10 tiene actividades pendientes",
  "path": "/quotations/10/accept",
  "timestamp": "2026-06-21T15:00:00.000Z"
}
```

<Warning>
  Accepting a quotation closes the linked lead atomically. If you receive a `409 Conflict`, it means two requests raced to change the quotation state simultaneously, or the lead has unresolved pending activities. Fetch the current quotation state with `GET /quotations/:id` and verify whether it was already accepted before deciding whether to retry.
</Warning>

## Handling 409 Conflict

A `409 Conflict` response can mean one of two things: two requests attempted to change the quotation state simultaneously, or the linked lead has unresolved pending activities. If two requests try to finalize the same quotation simultaneously, one will succeed and the other will receive a `409 Conflict`. This protects against double-processing ŌĆö for example, preventing a quotation from being both accepted and rejected at the same time.

To recover:

1. Call `GET /quotations/:id` to retrieve the current state of the quotation.
2. Check whether the desired outcome has already been applied (e.g., the quotation is already `ACEPTADA`).
3. If the state is already what you wanted, treat the operation as successful and move on.
4. If the state is different from what you expected, evaluate the current state and decide whether to retry or surface an error to the user.

<Tip>
  Build your integration to be idempotent around lifecycle transitions. Before calling `/accept` or `/reject`, check the current `estado` on the quotation ŌĆö if it is already terminal, skip the call entirely.
</Tip>
