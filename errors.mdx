The Bioactiva CRM API uses standard HTTP status codes to signal success and failure. Every `4xx` or `5xx` response includes a JSON body that describes what went wrong. Depending on which module produced the error, the body follows one of two shapes — a **standard shape** or an **extended shape**.

---

## Standard error shape

Most API modules return errors in this format:

```json
{
  "statusCode": 400,
  "message": ["correo must be an email"],
  "error": "Bad Request"
}
```

| Field | Type | Description |
|---|---|---|
| `statusCode` | `number` | The HTTP status code |
| `message` | `string \| string[]` | A human-readable description of the error. Can be a single string or an array of strings when multiple validation constraints fail simultaneously |
| `error` | `string` | The HTTP status reason phrase (e.g. `"Bad Request"`, `"Unauthorized"`) |

---

## Extended error shape

For activities, leads, and quotations, error responses include additional diagnostic fields:

```json
{
  "statusCode": 404,
  "error": "ActivityNotFoundException",
  "kind": "NotFound",
  "message": "Actividad con id 99 no encontrada",
  "path": "/activities/99",
  "timestamp": "2026-05-31T10:30:00.000Z"
}
```

| Field | Type | Description |
|---|---|---|
| `statusCode` | `number` | The HTTP status code |
| `error` | `string` | A stable error identifier for this error type (e.g. `"ActivityNotFoundException"`) |
| `kind` | `string` | Semantic category of the error: `"Validation"`, `"NotFound"`, or `"Conflict"` |
| `message` | `string \| string[]` | Human-readable description of the error |
| `path` | `string` | The request path that produced the error |
| `timestamp` | `string` | ISO 8601 timestamp of when the error occurred |

<Tip>
  Use the `error` field as a stable identifier in your error-handling code, rather than parsing the `message` string, which may change between releases.
</Tip>

---

## Status codes

| Code | Meaning | Common causes |
|---|---|---|
| `400` | Bad Request | Invalid request body, invalid token, passwords don't match, invalid state transition |
| `401` | Unauthorized | Missing or expired access token, invalid credentials |
| `403` | Forbidden | Authenticated user lacks the required role |
| `404` | Not Found | Resource does not exist or has been soft-deleted |
| `409` | Conflict | Duplicate state, concurrent modification, admin self-disable |

---

## Validation errors

When a request body fails validation, the API returns `400 Bad Request` and the `message` field is an **array** listing every constraint that was violated:

```json
{
  "statusCode": 400,
  "message": [
    "correo must be an email",
    "password should not be empty",
    "property extraField should not exist"
  ],
  "error": "Bad Request"
}
```

Read through every entry in the array before retrying — the request will continue to fail until all listed constraints are satisfied.

<Note>
  The API rejects unknown properties. If you send a field that is not recognised, the request will fail with a message like `"property <fieldName> should not exist"`. Remove any unexpected fields before retrying.
</Note>

---

## 401 vs 403

<Note>
  **401 Unauthorized** means you are not authenticated — either you did not include an `Authorization: Bearer` header, your access token has expired, or the token is malformed. Refresh your token and retry.

  **403 Forbidden** means you are authenticated but your role is not permitted to perform the action. For example, a `TRABAJADOR` calling an `ADMINISTRADOR`-only route receives `403`. Refreshing your token will not help — you need an account with the appropriate role.
</Note>

## Concurrent modification (409)

Some resources — notably quotations — use **optimistic locking** to guard against race conditions. If two requests read the same record and then both attempt to modify it simultaneously, the second write detects that the state has already changed and returns `409 Conflict`:

```json
{
  "statusCode": 409,
  "error": "CotizacionConflictException",
  "kind": "Conflict",
  "message": "La cotización con id 1 ya fue procesada por otra operación",
  "path": "/quotations/1/accept",
  "timestamp": "2026-06-01T10:30:00.000Z"
}
```

When you receive a `409`, **do not retry the original request blindly**. Instead:

1. Fetch the current state of the resource with a `GET` request.
2. Inspect the updated state to determine whether your intended action is still valid.
3. If it is still valid, resend your request against the freshly fetched state.

<Warning>
  A `409` on `PATCH /quotations/:id/accept` or `PATCH /quotations/:id/reject` means another operation already processed that quotation — and the linked lead's state may have already changed to `CIERRE_CON_VENTA` or `CIERRE_SIN_VENTA`. Always re-fetch before deciding how to proceed.
</Warning>

A `409` can also occur in the users module when an administrator attempts to disable their own account, or when you try to set a resource to the state it is already in (e.g. enabling an already-active user).
